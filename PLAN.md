# LLM Blueprint — 기획서

> "내 질문이 답이 되기까지" — 사용자가 고른 질문 하나가 스크롤을 따라
> 토큰 → 숫자 → 벡터 → 어텐션 → 확률 → 답변으로 변해가는 과정을
> **기계 설계도(블루프린트)** 스타일로 보여주는 교육용 스크롤 웹페이지.

---

## 1. 콘셉트

- transformer-explainer가 "아키텍처 다이어그램"이 주인공이라면, 이 페이지는 **입력한 문장 자체가 주인공**. 문장이 부품처럼 분해·가공·조립되어 답변이 되는 과정을 1인칭으로 따라간다.
- 세계관: **"LLM의 설계 도면을 펼쳐본다"**. 페이지 전체가 한 장의 거대한 청사진이고, 스크롤 = 도면을 따라 내려가며 공정을 견학하는 것.
- 교육용: 각 공정마다 캡션 카드(기본 설명 2~3문장 + "더 깊이" 접기에 수식·용어). UI 언어는 한/일/영 토글.

## 2. 확정 사항

| 항목 | 결정 |
|---|---|
| 설명 깊이 | 기본(비유 중심) + 씬별 "더 깊이" 접기(QKV·softmax 등 실제 용어/수식) |
| 구현 형태 | **멀티 파일 정적** (빌드 도구 없음, ES 모듈) — GitHub 저장소 가독성 + GitHub Pages 직배포 |
| 스타일 | **블루프린트** (청사진 도면: 딥블루 배경, 화이트/시안 선화, 그리드, 치수선, 스탬프) |
| 데이터 전략 | 엔진은 범용(스키마만 알면 어떤 질문이든 렌더), 콘텐츠는 프리셋 9문항(한/일/영 ×3)을 오프라인에서 실제 토크나이저로 베이크 |
| 로컬 실행 | `index.html` 더블클릭으로 즉시 실행 — ES 모듈 대신 일반 `<script>` 로딩을 채택해 `file://` 제약을 제거 (서버 불필요) |
| 커스텀 질문 | v1에 포함 — `js/config.js`의 `enableCustomInput` 또는 URL `?custom=1`로 활성화. 호스팅 페이지는 기본 OFF(프리셋만 노출) |

## 3. 사용자 여정

1. 랜딩(도면 표제란): 타이틀 + 우상단 언어 토글 + 질문 선택 패널(9개 프리셋)
2. 질문 선택 → 채팅 입력창에 타이핑 애니메이션 → 전송
3. 이후 **스크롤이 곧 시간축**. 내리면 공정 진행, 올리면 되감기 (GSAP scrub)
4. 답변이 채팅 버블로 완성 → 전체 공정 전도(全圖) 미니맵 → 다른 질문으로 재실행

## 4. 씬 스토리보드 (12 scenes)

각 씬 공통 레이아웃: 화면 좌하단 **캡션 카드**(도면 주석 스타일: 씬 번호 `SEC.02`, 제목, 본문 2~3문장, ▾더 깊이 접기), 중앙~우측 **비주얼 스테이지**(pin + scrub).

| # | 씬 | 비주얼 (블루프린트 모티프) | 애니메이션 | 더 깊이 |
|---|---|---|---|---|
| 0 | 표지 / 질문 선택 | 도면 표제란(title block), 도면 번호 = 질문 ID | 그리드 페이드인, 질문 카드 hover 시 치수선 | LLM이란 무엇인가 한 줄 정의 |
| 1 | 전송 | 채팅 입력창 → 문장이 떠올라 도면 중앙의 긴 스트립으로 고정, 클램프(고정 기호) 표시 | 타이핑 → 전송 → 스트립 안착 | 컨텍스트 윈도우 개념 |
| 2 | 토큰화 | 스트립 위에 **절단선(점선+가위 마크)** 이 그어지고 조각으로 분리. 각 조각 아래 부품 태그(토큰 ID) | 절단선 드로잉 → 조각이 벌어짐 → ID 태그 스탬프 | BPE, 어휘 사전. **같은 질문의 한/일/영 토큰 수 비교 오버레이**(sky 질문 한정) |
| 3 | 임베딩 | 각 부품이 거대한 **룩업 테이블 도면**(임베딩 행렬 단면)을 통과 → 뒷면이 숫자 배열(가로 히트맵 스트립, 시안 농담) | 카드 플립 → 숫자 롤링 → 스트립 정렬 | 벡터 = 의미의 좌표. 보조: 아이소메트릭 미니 산점도(비슷한 단어끼리 근접) |
| 4 | 위치 인코딩 | 각 스트립에 **위치 좌표 스탬프**(#0, #1…)와 파형 오버레이가 겹쳐 찍힘 | 파형이 좌→우로 스윕하며 각인 | sin/cos 또는 RoPE 한 줄 소개 |
| 5 | 어텐션 | 스트립들이 세로로 정렬, 그 사이를 **배선도(wiring) 곡선**이 연결. 선 굵기 = 가중치, 핵심 쌍에 주석 플래그 | 배선이 순차 드로잉, 가중치 라벨 카운트업 | Q·K·V — "누구를 볼까(Q·K), 무엇을 가져올까(V)", 멀티헤드 = 색상 레이어 토글 |
| 6 | FFN | 각 스트립이 개별적으로 **확장→압축 필터 유닛**(기계 단면도: 넓어졌다 좁아지는 덕트) 통과 | 스트립이 유닛 통과하며 색 분포 변화 | 확장비 4x, 활성함수, "지식이 저장되는 곳" 가설 |
| 7 | 레이어 스택 | **입체감 하이라이트.** 도면이 아이소메트릭으로 기울고, 방금 본 공정(5+6)이 하나의 층 = 카메라가 N층 단면을 수직 하강 | 층 반복 통과, 층 카운터 `LAYER 07/32`, 스트립 색이 점점 "의미"로 수렴 | 층 수·파라미터 규모 비교표(소형~대형 모델) |
| 8 | 출력 확률 | 마지막 스트립이 **게이지 패널**에 연결 → 어휘 후보 top-5가 막대 게이지로, 확률 % 라벨 | 게이지 스윕 → softmax 정규화(막대 합=100%) | logits→softmax 수식, temperature 슬라이더(개념 시연) |
| 9 | 샘플링 & 루프 | 게이지에서 하나 추첨(스탬프 "SELECTED") → 새 토큰이 **순환 컨베이어 배관**을 타고 입력 스트립 끝에 부착 → 파이프라인 아이콘 재점등 | 스크롤 진행마다 답 토큰 1개씩 스트리밍(루프 3~4회는 풀 애니메이션, 이후 빨리감기) | 오토리그레시브, greedy vs sampling |
| 10 | 완성 | 답 부품들이 조립되어 채팅 버블로. **EOS 토큰에 "승인(APPROVED)" 도장** → 출력 종료 | 조각들 슬라이드 결합 → 버블 확정 | EOS/stop 토큰, 스트리밍 출력의 정체 |
| 11 | 전도(全圖) | 파이프라인 전체 미니맵(공정 0~10 축소 도면), 클릭 시 해당 씬으로 점프. "다른 질문으로 재실행" 버튼 | 전체 도면이 한눈에 펼쳐짐 | "학습(training)은 별도 도면" 안내 + 참고 링크 |

**분량 감각**: 씬당 100~200vh, 전체 약 15~20 스크린 분량. 우측 가장자리에 공정 진행 레일(현재 씬 하이라이트, 클릭 점프).

## 5. 인터랙션 설계

- **UI 언어 토글**(우상단 고정, KO/JA/EN): 캡션·버튼·라벨 즉시 전환. 질문 언어와 독립 — 일본어 질문을 한국어 설명으로 볼 수 있음.
- **질문 전환**: 씬 11 버튼 또는 진행 레일 최상단 → 인트로로 스크롤 복귀 + 상태 리셋.
- **더 깊이 접기**: 캡션 카드 하단 ▾. 열어도 스크롤 타임라인에 영향 없음(카드 내부 스크롤).
- **reduced motion**: `prefers-reduced-motion` 시 scrub 애니메이션 대신 씬 단위 스냅 + 정적 다이어그램.
- **모바일**: 씬 구성 유지, 아이소메트릭 각도 축소, 캡션 하단 시트화. 1차 타깃은 데스크톱.

## 6. 데이터 설계 — "엔진은 범용, 콘텐츠는 프리셋"

### 6.1 QuestionData 스키마 (`data/questions.json`)

```jsonc
{
  "id": "sky-ko",
  "lang": "ko",
  "question": "하늘은 왜 파란가요?",
  "tokens": [ { "t": "하늘", "id": 14307 }, ... ],      // 실제 토크나이저(o200k_base) 결과
  "answer": "짧은 파장의 파란 빛이 공기 분자에 더 많이 흩어지기 때문이에요. ...",
  "answerTokens": [ ... ],                               // 동일 방식
  "attention": {
    "highlights": [ { "from": 4, "to": 0, "w": 0.8, "note": {"ko":"...","ja":"...","en":"..."} } ],
    "procedural": "distance-decay"                       // 나머지 배선은 규칙 생성
  },
  "steps": [                                             // 답변 토큰별 top-k (씬 8~9용)
    { "topK": [ {"t":"짧은","p":0.42}, {"t":"파란","p":0.21}, ... ], "chosen": 0 }
  ],
  "notes": { "ko": "이 질문의 관전 포인트", "ja": "...", "en": "..." }
}
```

엔진은 이 스키마만 알면 어떤 질문이든 렌더 → **새 질문 추가 = JSON 항목 1개 추가**.

임베딩 히트맵·의미공간 좌표 등 순수 시각화용 수치는 데이터에 넣지 않고 **토큰 ID를 시드로 절차 생성**(같은 토큰은 항상 같은 무늬 → 재현성·교육적 일관성 확보, 데이터 파일 경량 유지).

### 6.2 프리셋 9문항

공통 1문항은 3개 언어에 모두 배치 → **씬 2에서 언어별 토큰 수 비교**라는 교육 포인트 확보.

| 언어 | 질문 |
|---|---|
| KO | 하늘은 왜 파란가요? / 김치는 왜 시어지나요? / AI는 어떻게 글을 쓰나요? |
| JA | 空はなぜ青いの？ / 猫はなぜ喉を鳴らすの？ / AIはどうやって文章を書くの？ |
| EN | Why is the sky blue? / How do planes fly? / How does AI write text? |

- 답변은 2~3문장, **답변 토큰 20~30개**로 제한(씬 9 루프 길이 관리).
- "AI는 어떻게 글을 쓰나요?"는 페이지 자신을 설명하는 메타 질문 — 마무리로 재미 요소.

### 6.3 생성 파이프라인 (`tools/generate_data.py`)

오프라인 Python 스크립트로 베이크, 저장소에 포함해 재현 가능하게:

1. 질문·답변 텍스트 정의 → `tiktoken`(o200k_base)으로 실제 토큰 분해 + ID 추출
2. top-k 후보: 정답 토큰 + 그럴듯한 오답 후보를 수작업 정의, 확률은 휴리스틱 분배
3. 어텐션: 핵심 쌍(예: 대명사→선행사, "왜"→서술어)만 수작업 지정, 나머지는 규칙
4. `questions.json` 출력

> 어텐션·확률은 실측값이 아니므로 페이지에 "개념적 재구성" 고지 필수 (11절 참조).

## 7. 기술 아키텍처

### 7.1 파일 구조

```
LLM_Explain/
├── index.html
├── css/
│   ├── blueprint.css        # 디자인 토큰(색·선·그리드·스탬프 클래스)
│   └── scenes.css           # 씬별 레이아웃
├── js/
│   ├── main.js              # 부트스트랩, 상태(선택 질문·언어)
│   ├── engine/
│   │   ├── director.js      # GSAP ScrollTrigger 마스터 타임라인, 씬 등록·pin 관리
│   │   ├── stage.js         # DOM/SVG/Canvas 레이어 헬퍼
│   │   └── i18n.js          # 언어 토글, 캡션 바인딩
│   └── scenes/

└── README.md                # 실행법(python -m http.server), 스크린샷, 고지
```

### 7.2 기술 선택

- **스크롤**: GSAP 3 + ScrollTrigger (cdnjs). pin + scrub으로 씬별 타임라인을 마스터에 연결. 되감기 무료 지원.
- **렌더링**: SVG(선화·배선·치수선 — 블루프린트와 최상 궁합, `stroke-dashoffset` 드로잉 애니메이션) + CSS 3D transform(아이소메트릭 기울임, 씬 7 층 하강) + Canvas(숫자 파티클 등 대량 요소만). **Three.js 미사용** — 도면 스타일엔 과하고 무거움. 필요해지면 씬 7만 교체(백로그).
- **폰트**: JetBrains Mono(숫자·라벨, 도면 느낌) + Noto Sans KR/JP(본문). Google Fonts + 시스템 폴백.
- **성능**: 토큰 조각은 DOM/SVG 노드 ≤ 40개, transform/opacity만 애니메이션(GPU 합성). 60fps 목표.
- **의존성**: GSAP 하나. 프레임워크 없음(Vanilla ES modules).

## 8. 비주얼 디자인 시스템 (블루프린트)

- **팔레트**: 배경 `#0D2B45`(청사진 블루)→`#081C30` 그라데이션, 주 선색 `#EAF6FF`(화이트), 강조 `#7FDBFF`(시안), 경고·스탬프 `#FFB454`(앰버), 확률·히트맵은 시안 농담 스케일.
- **모티프 규칙**: 모든 도형은 선화(면 채움 최소), 그리드 종이 배경(major/minor grid), 치수선·지시선·절단선 기호, 모서리 등록 마크, 요소 등장 = 선이 그려지는 드로잉 애니메이션, 확정 = 스탬프 찍힘.
- **입체감**: 아이소메트릭 투영(CSS `rotateX`+`rotateZ`)과 그리드 원근, 레이어별 패럴랙스. 씬 7에서 최대치.
- **진행감**: 씬이 진행될수록 배경 색온도가 미세하게 상승(차가운 입력→따뜻한 답변), 답변 완성 시 앰버 하이라이트.
- **타이포**: 캡션 제목 크게, 수치는 모노스페이스, 도면 주석풍 대문자 라벨(`TOKENIZER — SEC.02`).

## 9. 교육 카피 구조

- `captions.json`: 씬별 `{ title, body, deep }` × ko/ja/en + UI 문자열(버튼·토글·고지).
- 톤 가이드: 본문은 **비유 1개 + 정확한 용어 1개 병기**(예: "단어 조각 = 토큰"). deep은 1문단, 수식은 1줄 이내.
- 카피 초안은 M3에서 3개 언어 동시 작성(한국어 원문 → 일/영 번역·감수).

| 마일스톤 | 내용 | 완료 기준 |
